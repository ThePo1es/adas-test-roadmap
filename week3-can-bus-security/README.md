# Week 3 — 실물 CAN 버스 구축 + 스니핑/스푸핑 + 안전규격

**목표:** 아두이노 2대 + 라즈베리파이로 **진짜 CAN 버스**를 만들고, 그 위에서 통신·분석·공격을 한다. 동아리(차량보안+임베디드)의 핵심 주간.
**준비물:** MCP2515 CAN 모듈 3개, 아두이노 2대, 라즈베리파이, 120Ω 종단저항. 배선은 `../HARDWARE.md` 참고.

---

## Day 1 (월) — CAN 개념 + 첫 노드 송신

| 시간 | 할 일 |
|------|------|
| 0:00~1:30 | CAN 개념: 프레임 구조(ID/DLC/Data/CRC), ID 중재(작을수록 우선), 차동신호(CAN_H/L), 종단저항 120Ω이 왜 2개인지 |
| 1:30~2:30 | 아두이노 #1에 MCP2515 배선(SPI, `../HARDWARE.md`). `coryjfowler/MCP_CAN_lib` 설치. **크리스털 8/16MHz 확인**(CAN_500KBPS 설정에 영향) |
| 2:30~4:30 | 아두이노 #1이 0.1초마다 카운터 값을 CAN 프레임(ID 0x100)으로 송신. 송신 성공 여부 시리얼 확인 |
| 4:30~5:00 | 커밋(배선 사진) |

**완료 기준:** 노드 #1이 CAN 프레임을 실제로 쏜다(계측기 없으면 Day2 수신으로 확인).
**실패 주의:** 크리스털 주파수 오설정(제일 흔한 함정), CS핀 불일치, 종단저항 없음/1개 → 통신 불가.

## Day 2 (화) — 두 번째 노드 수신 (버스 완성)

| 시간 | 할 일 |
|------|------|
| 0:00~2:00 | 아두이노 #2에 MCP2515 배선, CAN_H↔CAN_H / CAN_L↔CAN_L로 두 노드 연결, 양 끝 120Ω |
| 2:00~4:00 | 노드 #2가 ID 0x100 수신 → 값 시리얼 출력. 두 보드가 실제로 통신! |
| 4:00~4:30 | 통신 안 되면 디버깅 체크리스트(크리스털/속도/배선/종단) 하나씩 |
| 4:30~5:00 | 커밋 |

**완료 기준:** #1이 보낸 카운터가 #2 시리얼에 그대로 찍힌다.

## Day 3 (수) — 라즈베리파이를 버스에 연결 (분석 장비)

| 시간 | 할 일 |
|------|------|
| 0:00~1:30 | Pi에 MCP2515 배선 + `config.txt`에 `dtoverlay=mcp2515-can0,...` 추가 → 재부팅 → `ip link set can0 up type can bitrate 500000` |
| 1:30~2:30 | `sudo apt install can-utils`. **`candump can0`로 아두이노들의 트래픽을 스니핑** — 실차 진단의 축소판 |
| 2:30~4:00 | `cansend can0 100#0102030405060708`로 Pi가 직접 프레임 송신 → 아두이노 #2가 받는지 확인. `cangen`으로 부하 생성 |
| 4:00~5:00 | candump 로그 저장 + 커밋 |

**완료 기준:** Pi에서 아두이노 트래픽이 candump에 실시간으로 흐른다.

## Day 4 (목) — DBC 디코딩 + 신호 스케일링

| 시간 | 할 일 |
|------|------|
| 0:00~1:30 | DBC 파일 개념(메시지/신호/start bit/scale/offset). opendbc(comma.ai)에서 실차 DBC 구경 |
| 1:30~3:00 | 우리 버스용 DBC 직접 작성: ID 0x100 = "VehicleSpeed", scale 0.01. `cantools`로 candump 로그 디코딩 → 물리값 |
| 3:00~4:30 | 아두이노 #1을 "거리/차속 방송 ECU"로 개조(Week2 초음파 결합): 실제 거리를 CAN으로 방송 → Pi에서 디코딩 |
| 4:30~5:00 | 커밋 |

**완료 기준:** 손을 센서에 대면 candump/cantools에 거리값이 물리단위로 변한다.

## Day 5 (금) — CAN 보안: 스푸핑 & 리플레이 (동아리 핵심)

| 시간 | 할 일 |
|------|------|
| 0:00~1:00 | 왜 CAN은 취약한가: 암호화·송신자 인증 없음, 브로드캐스트. 스푸핑/리플레이/DoS(버스 점유) 개념 |
| 1:00~3:30 | **과제 1: 공격 실습** — Pi에서 ① candump로 정상 거리 프레임 캡처 → ② `canplayer`로 리플레이 → ③ `cansend`로 **가짜 "먼 거리" 프레임을 고빈도 주입**해 정상 프레임을 덮어쓰기 |
| 3:30~4:30 | 관찰: 수신 ECU가 가짜값을 믿는지 확인 → `src/attack_log.md`에 공격 시나리오/결과 기록. **이게 TARA의 실증** |
| 4:30~5:00 | 커밋 |

**완료 기준:** 가짜 프레임 주입으로 수신 노드의 판단을 바꿀 수 있음을 실측으로 보였다.
**방어 맛보기(선택):** 메시지 카운터+간이 체크섬(SecOC 개념)을 프레임에 넣어 리플레이 탐지 시도.

## Day 6 (토) — 안전·보안 규격 정리

| 시간 | 할 일 |
|------|------|
| 0:00~1:30 | ISO 26262(기능안전): V모델, ASIL(S×E×C). **시험 직무는 V모델 오른쪽** |
| 1:30~2:30 | ISO 21448(SOTIF): 고장이 아닌 성능한계·오사용 / ISO/SAE 21434(사이버보안): TARA. 공고에 셋 다 관련 |
| 2:30~4:30 | **과제 2: `src/safety_summary.md`** — 세 규격 각 10줄 + "우리 AEB 버스에 적용하면 필요한 시험" 각 2가지(Day5 공격을 21434 사례로 연결) |
| 4:30~5:00 | 커밋 |

**완료 기준:** 내가 한 CAN 스푸핑 실습을 ISO 21434 언어(위협/리스크)로 설명할 수 있다.
