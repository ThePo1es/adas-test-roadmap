# Week 4 — 캡스톤: 물리 AEB 테스트베드 + 시험 자동화 + 보안 시험

**목표:** 3노드 CAN 버스 위에서 시험 직무의 전 사이클(요구사양→Test Case→시험 실행→자동판정→리포트)을 **실물로** 한 바퀴. 여기에 CAN 보안 시험까지.
**이 폴더가 포트폴리오의 메인.**

## 최종 시스템

- **아두이노 #1 (Sensing ECU):** HC-SR04로 전방 거리 측정 → CAN ID 0x100으로 방송
- **아두이노 #2 (AEB ECU):** 거리 수신 → 접근속도/TTC 계산 → 임계 이하 시 모터 정지 + 부저 + LED
- **라즈베리파이 (Test Bench):** can-utils/python-can으로 ①시험 시나리오 실행 ②전 프레임 로깅 ③PASS/FAIL 자동판정 ④가짜 프레임 주입 보안 시험

## Day 1 (월) — 요구사양 작성

| 시간 | 할 일 |
|------|------|
| 0:00~1:00 | 좋은 요구사항 = 측정가능·검증가능·모호어 금지 |
| 1:00~4:00 | `requirements.md` — AEB 요구사항 6개(ID 부여). 예: "REQ-01: 전방 장애물 15cm 이내에서 100ms 내 모터 정지" / 경고 타이밍 / 미작동 조건(오작동 방지) / **REQ-보안: 비정상 고빈도 거리 프레임 주입 시 시스템이 안전측(제동 유지)으로 동작** |
| 4:00~5:00 | 각 REQ가 우리 하드웨어로 시험 가능한지 셀프체크 + 커밋 |

**완료 기준:** 6개 REQ 전부 수치+측정방법 존재, 보안 요구 1개 포함.

## Day 2 (화) — Test Case 설계

| 시간 | 할 일 |
|------|------|
| 0:00~1:00 | TC 양식: ID/연계REQ/사전조건/입력(거리·접근속도)/절차/합격기준(정지여부·반응시간·최종거리) |
| 1:00~4:00 | `test_cases.md` — 정상 시나리오(정지 장애물/접근 장애물/천천히 접근)×거리 3단계 + 경계값(임계 바로 위/아래) + **보안 TC(스푸핑/리플레이)** 2개 = 최소 10 TC, REQ 추적표 포함 |
| 4:00~5:00 | 커밋 |

**완료 기준:** 10개 TC, 전부 REQ 연결, 합격기준 수치화, 보안 TC 포함.

## Day 3~4 — 테스트베드 구현

| 시간 | 할 일 |
|------|------|
| Day3 0:00~2:30 | 아두이노 #1 Sensing ECU 완성(초음파→CAN 방송), Pi candump로 검증 |
| Day3 2:30~5:00 | 아두이노 #2 AEB ECU: 거리 수신 → 이전값과 차분으로 접근속도 → 간이 TTC → 제동. `src/aeb_ecu.ino` |
| Day4 0:00~2:30 | Pi 시험 벤치 `src/test_bench.py`(python-can): 프레임 실시간 로깅 + 반응시간 측정(거리 임계 통과~모터정지 CAN 관측까지) |
| Day4 2:30~4:00 | **검증:** 손계산/스톱워치와 Pi 측정 반응시간 비교 → 측정 신뢰성 확인 |
| Day4 4:00~5:00 | 파라미터 스터디: 임계거리/TTC 바꿔가며 오작동↔지연 트레이드오프 기록. 커밋 |

**완료 기준:** 장애물을 들이대면 실제 모터가 서고, Pi가 반응시간을 자동 측정한다.
